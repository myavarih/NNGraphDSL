# Architecture

## Pipeline overview

```
.nng source file
       │
       ▼  antlr4.FileStream → NNGraphLexer → CommonTokenStream
       │
       ▼  NNGraphParser.program() → parse tree (ANTLR4 context tree)
       │
       ▼  ParseTreeWalker + NNGraphCustomListener
       │    exit* methods fire bottom-up; build node/edge model;
       │    run 5 validation passes on exitGraph_block
       │
       ▼  CodeGenerator.load_from_listener(listener)
       │    _compute_graph()   — build adj lists, Kahn topo sort
       │    _assign_vars()     — assign Python var name per node
       │    _emit_init()       — generate __init__ lines
       │    _emit_forward()    — generate forward() lines
       │    _assemble()        — join into complete Python source
       │
       ▼  .py output file (or stdout)
```

Entry point: `compile_nng()` in `main.py:14`.

## Data flow between stages

### Stage 1 → 2 (lexer → parser)

`FileStream` reads bytes from the `.nng` file. `NNGraphLexer` produces a token
stream. `NNGraphParser.program()` builds a parse tree — a tree of `*Context`
objects, one per grammar rule match.

### Stage 2 → 3 (parse tree → listener)

`ParseTreeWalker.walk(listener, parse_tree)` traverses the tree depth-first.
For each node it calls `enter*` (before children) then `exit*` (after children).
The listener uses only `exit*` methods, so by the time any method fires, all child
contexts have already been processed and their `.pvalue`/`.ptype` attributes are set.

The listener builds three structures:
- `listener.nodes` — `dict[id, {type, params, line}]`
- `listener.edges` — `list[{src, dst, label, line}]`
- `listener.config` — `dict[key, value]`

At `exitGraph_block`, the listener runs 5 validation passes over the completed graph.

### Stage 3 → 4 (listener → codegen)

`CodeGenerator.load_from_listener(listener)` copies the three structures above plus
`model_name`, `input_id`, `input_shape`, and `output_id`.

`CodeGenerator.generate()` runs four internal passes in fixed order:
1. `_compute_graph()` — builds adjacency lists and topological order
2. `_assign_vars()` — assigns a Python variable name to each node's output tensor
3. `_emit_init()` + `_emit_forward()` — produce line lists
4. `_assemble()` — joins everything into the final Python string

### Stage 4 → output

`compile_nng()` writes the string to a file (if `-o` is given) and/or prints it
to stdout (if `--print` is given). The generated file is standalone Python; it does
not import any compiler code.

## Listener pattern

ANTLR4 provides two tree-walking patterns: listener and visitor. This compiler uses
the listener pattern. `ParseTreeWalker` drives the walk; the listener's `exit*`
methods are called automatically without any return values needed.

This suits the use case because the listener accumulates state into `self.nodes`
and `self.edges` across many rule exits, rather than computing a return value
per subtree. The EVM ANTLR project in `PycharmProjects/Antlr` uses the same
pattern.

---

## Context attribute propagation

ANTLR4 context objects (`*Context`) are plain Python objects. The listener attaches
custom attributes to them at `exit*` time:

- `exitShape_expr` sets `ctx.pvalue = tuple(ints)` and `ctx.ptype = tuple`
- `exitValue` reads child contexts and sets its own `ctx.pvalue` / `ctx.ptype`
- `exitParam` reads `ctx.value().pvalue` and stores it on `ctx.param_value`
- `exitLayer_expr` reads all `param()` children and builds `ctx.params`
- `exitNode_decl` reads `ctx.layer_expr().layer_type` and `ctx.layer_expr().params`

This bottom-up propagation means each rule can assume its children are fully
evaluated when its own `exit*` fires.

## Error reporting

Two functions in `custom_listener.py`:

```python
def error(msg, ctx):           # line 37
    print(f"[line {line}:{col}] SemanticError: {msg}", file=sys.stderr)
    raise SystemExit(1)

def warn(msg, ctx):            # line 44
    print(f"[line {line}] Warning: {msg}", file=sys.stderr)
```

Errors abort immediately. Warnings continue the walk.

Post-graph-walk errors (raised in `exitGraph_block` validation methods) do not have
a live `ctx.start.line`. A `_FakeCtx` inner class carries the line number from the
node's original declaration, keeping the error format consistent.
